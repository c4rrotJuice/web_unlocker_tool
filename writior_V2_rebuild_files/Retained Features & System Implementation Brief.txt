Retained Features & System Implementation Brief
Purpose
This brief instructs the agent how to rebuild Writior v2 while preserving the valuable capabilities of the old product and adapting them to the new architecture, database, and extension-first workflow.

The previous database tables and legacy implementation patterns are not to be reused.
All features must be rebuilt against the new canonical schema and modernized flows.

The rebuild must produce a clean production-grade system that preserves functionality but removes:

legacy database coupling

duplicate logic between extension and backend

old unlock web interface

fragmented research entities

weak authentication boundaries

The final product must operate as a coherent research → writing system centered around:

extension capture

canonical research graph

editor composition workspace

Core Architectural Principles
1. Extension-first system
The browser extension becomes the primary entry point into Writior.

All research capture flows originate from the extension:

highlight text

capture quote

capture citation

create note

send to editor

The old web-based URL fetch unlock interface is removed.

Useful concepts from that system remain but are adapted to:

copy assist

selection capture

research attribution

unlock analytics

2. Canonical research graph
All research entities must exist in a single normalized graph:


Source
   ↓
Citation Instance
   ↓
Quote
   ↓
Note
   ↓
Document
Each entity has a clear responsibility:

Entity	Meaning
Source	origin webpage/book/article
Citation Instance	user-owned formatted citation
Quote	captured evidence
Note	synthesis / commentary
Document	final composition

Extension and editor must write to the same graph services.

3. Canonical backend authority
The backend must be the single authority for:

entitlements

research entities

documents

relations

analytics

billing capability checks

The extension must never be the source of truth.

All features must use the new canonical schema only.

2. Session architecture
Web session model
For app.writior.com, use Supabase-authenticated browser sessions and require authenticated frontend API requests to send a bearer token.

Backend APIs must:

accept Authorization: Bearer <token>,

verify the token against Supabase,

resolve the user id from the verified token,

load canonical account/capability state from canonical tables.

Forbidden legacy pattern
Do not:

mirror access tokens into JS-readable cookies,

trust browser-set auth cookies as equivalent to verified bearer identity,

preserve the old wu_access_token strategy. 
system audit


Optional server-managed cookie mode
If a cookie-backed app session is introduced later, it must be:

HttpOnly

Secure

SameSite=Lax or stricter

scoped to app.writior.com

independently managed and rotatable

But the default v2 implementation should remain: verified bearer-token API auth, not raw cookie auth.

10. Request hardening
The current audit explicitly notes:

no CSRF middleware,

no generic rate-limit middleware,

only baseline security headers,

route/service-local throttling. 
system audit


That must be hardened.

Required hardening measures
CORS
Implement strict production CORS allowlists.

Allowed origins should be limited to:

https://app.writior.com

https://writior.com only where necessary

extension origins

local dev origins in non-production

No wildcard production CORS.

CSRF
If protected APIs rely exclusively on bearer-token auth and not cookies, CSRF exposure is significantly reduced.

Still:

add CSRF protection for any cookie-authenticated mutation endpoints,

do not allow mixed trust between cookies and bearer auth,

prefer token-authenticated API calls for SPA-like surfaces.

Security headers
Add/strengthen:

Content Security Policy

HSTS

X-Content-Type-Options: nosniff

Referrer-Policy: strict-origin-when-cross-origin

Permissions-Policy

frame protection via CSP frame-ancestors 'none' or equivalent

secure cache-control for sensitive auth pages where appropriate

Rate limiting
Implement generic middleware/service support for rate limiting and apply it especially to:

login/signup

password reset if added

OAuth callback abuse points if needed

handoff issue/exchange

extension usage-event endpoints

unlock endpoints

search/report endpoints

webhook endpoints where appropriate

Redirect/path validation
Validate every redirect path and next destination.
Only allow:

internal relative paths,

normalized app paths,

known auth callback completions.

Reject:

external URLs,

protocol-relative redirects,

injected malformed paths.

Logging hygiene
Never log:

access tokens

refresh tokens

handoff code values

raw session payloads

sensitive provider secrets

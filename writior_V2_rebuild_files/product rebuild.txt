My recommendation in one line
Build Writior Core v2 on the same stack, but with:

one canonical auth/account model

one canonical research schema

one centralized entitlement service

one clean API/service/repository structure

one slimmer editor runtime

one thinner extension sync layer

That gets you a stronger product without starting from scratch.

This module should own:

What I would redesign hard
These are the destabilized parts:

1. Session strategy
Stop mirroring auth into a JS-readable weak cookie as the main server trust boundary. Keep handoff, but tighten the browser auth model. The current audit flags this as a top risk. 
system audit


2. Route ownership
Notes/projects/tags should not live under extension.py when they are core product APIs. Citations, editor, and extension should call shared services, not each own persistence rules. 
system audit


3. Entitlements
No more extension tier cache as product truth. It can cache UI state, but the backend must own policy. 
product features


4. Unlock surface
Merge /view and /fetch_and_clean_page into one canonical unlock flow. The iframe/postMessage legacy path should go. 
product features


5. Reporting data
Rebuild reporting on canonical citation tables and canonical research entities only. No legacy citation counting. 
product features


6. Transaction boundaries
Relation-heavy writes should all use atomic RPC-backed or transaction-backed service methods. The audit is very clear that outside selected RPCs, many writes are still multi-step best effort.

Editor architecture: keep Quill, shrink the shell
I would keep Quill for v2.

So what does “fresh” actually mean here?
For your case, “fresh” should mean:

fresh schema

fresh service boundaries

fresh API contracts

fresh editor runtime

fresh extension contracts

But not:

new auth provider

new deployment stack

new secrets

unnecessary stack replacement

That’s the best balance.

My recommendation on execution style
Do not ask Codex:
“Rebuild the entire app from scratch in one go.”

Do ask Codex:
“Implement Phase 1 only, exactly to spec.”
Then:
“Implement Phase 2 only.”
Then:
“Implement Phase 3A–3D.”
And so on.

That gives you:

inspectable progress

easier rollback

cleaner diffs

fewer hallucinated assumptions

better quality

Best practical setup
Use the agent in separate high-discipline prompts, each with:

exact scope

exact files/modules expected

explicit non-goals

acceptance criteria

“do not touch” constraints

For example:

Prompt 1: schema + migrations

Prompt 2: app core + auth + entitlements

Prompt 3: unlock module

Prompt 4: research graph module

Prompt 5: workspace module

Prompt 6: extension adaptation

Prompt 7: cleanup + dead-code removal

That is much safer than one mega-prompt.

How to instruct the agent
When we get to extension reconstruction, the prompt should explicitly say something like:

Preserve the current extension UX patterns and visual interaction model wherever they are already effective. Rebuild internals, contracts, and data flow to align with the new canonical backend, but do not redesign the extension UI unless required by a clearly identified product or usability problem.

That line matters a lot, because otherwise the agent may “improve” things you actually like.
